## 🎯 Приоритеты (что чинить первым) — синтез completeness-критика

**1.** FIX THE SHIPPED .apk DEPENDENCIES (confirmed in 4 sections): add kmod-nft-socket and kmod-nft-tproxy (and review ucode-mod-fs / drop spurious jq) to APP_DEPENDS in scripts/build-apk.sh so it matches LUCI_DEPENDS and CLAUDE.md. Then extend test_makefile_deps.sh to guard the build-apk.sh list too — right now the only guard checks the Makefile, i.e. the path NOT used for the real package. This is a real prod break: tproxy/socket firewall rules on a clean install have no kernel-module backing.

**2.** SERIALIZE THE SERVICE LIFECYCLE (newly surfaced, systemic): introduce one mkdir-style lock (reuse nftables.uc's proven primitive) around the generate→sing-box check→procd restart→nft apply sequence, and route both cron jobs through a single wrapper so subscription.uc and nft-rulesets.uc cannot both stop+start sing-box on the same */15 tick while a UI restart/generate also races. Today nothing prevents two concurrent stop+start cycles from leaving the daemon down.

**3.** FIX THE HIGH-SEVERITY CORRECTNESS BUGS that make sing-box refuse to start or run wrong config: (a) selector/urltest group members not validated against valid outbounds → dangling reference breaks sing-box load (CODE-core, high, needs_adjustment); (b) route_rule default emitting action:"" when UCI action unset (CODE-builder, medium); (c) post_process implicit-ref scrub being dead code because it runs before config.route exists (CODE-core, medium). Each produces a config sing-box rejects or mis-applies.

**4.** CLOSE THE README/DOCS CORRECTNESS GAP that misleads installers: README claims noarch + OpenWrt 24.10 (reality: per-arch, 25.12) and promises a firewall-conflict warning/removal the .apk and install.sh do not actually perform (3× high in EXTRA). A user on fw3 following the README can end up with neither firewall correctly applied. Fix the README claims and either implement the documented conflict handling or delete the promise.

**5.** RECONCILE CLAUDE.md WITH CODE for the two stale-but-load-bearing facts: SINGBOX_MAX_SKIPS is 25 in code not 15 (high docs-drift, affects CI defense-in-depth understanding), and the APP_DEPENDS kmods claim. Since CLAUDE.md is gitignored and never CI-checked, also adopt the digest's suggestion to add a lightweight committed reference or a CI doc-lint for the few load-bearing constants (skip ceiling, dep lists, OpenWrt version) so this class of drift stops recurring.

**6.** RUN THE TESTS CI CLAIMS TO RUN: add a host job (node) for the 5 node-gated *_js* / browser-shaped test files that currently only SKIP in the VM, and add a CI step that runs scripts/regen-po.sh and fails on a non-empty .pot diff. Without these, version-gating UX, descriptor_form dynamic widgets, and i18n freshness have zero executing coverage despite having test files.

## 🔗 Сквозные темы (повторяются в нескольких разделах)

- Concurrency/atomicity is solved locally but not globally: publish_atomic protects the config file and the nft apply-lock protects the firewall, yet the end-to-end service lifecycle (generate→check→restart→nft) has no mutex, and two cron jobs + UI restart + UI generate all mutate it from separate processes. The project's atomicity discipline stops at the file boundary and does not extend to the daemon-restart sequence.
- Dependency-list divergence is the single most-confirmed defect, appearing independently in CI-build, CI-release, EXTRA, and CLAUDEMD: the shipped .apk (scripts/build-apk.sh APP_DEPENDS) omits kmod-nft-socket/kmod-nft-tproxy that the tproxy/socket firewall path needs and that LUCI_DEPENDS + CLAUDE.md both claim are present. Same root cause = two hand-synced dep lists with no guard catching the drift in the SHIPPING path (test_makefile_deps.sh guards Makefile, not the real apk).
- Documentation drifts from code in many independent places — CLAUDE.md (skip-threshold 25 vs 15, export_section -L shebang, APP_DEPENDS kmods), README (noarch/24.10 vs per-arch/25.12, firewall-conflict removal promise), docs/ (release.md → deleted run-docker.sh, protocol-coverage.md old lib/protocols/ path + missing inbounds). The recurring failure mode is human-maintained prose that nothing in CI verifies against the tree.
- CI does not actually execute several of its own guards: 5 node-gated JS test files SKIP in the VM and have no host job, the SKIP ceiling is a rising moving target, and i18n/po-coverage and parity-corpus completeness rely on tests whose real execution path is unconfirmed. Several 'guards' provide a false sense of coverage.
- Silent/degraded fallback without operator-visible signal recurs: generate.uc falling back to stale config on write failure, subscription meta sidecar going stale on a headerless refetch, profile-title base64 partial-decode masking, selector/urltest emitting empty/dangling member lists. The system prefers to keep running over surfacing that it is running degraded.

**1. [CI-release/DEP-1] Shipping .apk omits kmod-nft-socket / kmod-nft-tproxy (and ucode-mod-fs) — tproxy/socket firewall path has no dependency backing**  
`high` · bug · scripts/build-apk.sh:29 · усилие: low · ✅ подтверждено

- **Проблема:** APP_DEPENDS (the dependency list baked into the real per-arch .apk, which is the primary supported install path per CLAUDE.md) is `libc luci-base sing-box jq curl`. ## 5+. Пропущенные области (от completeness-критика)

Это области, которые 10 scoped-агентов покрыли слабо или не покрыли — критик выделил их отдельно как ответ на п.5 «дополнительные пункты».

**1. Service-lifecycle concurrency: no lock serializes generate.uc / init.d start+stop / init.d reload across the four trigger paths**  
- _Почему важно:_ Only the nftables apply path is serialized (lock-DIR /tmp/singbox-ui/.apply.lock). The full lifecycle (generate.uc → publish_atomic config → sing-box check → procd start/stop → nft apply) has NO mutex. Four triggers can interleave: rpcd `restart` (etc/init.d/singbox-ui:113 reload_config = stop;start) and `generate` (rpcd singbox-ui:185), the boot path (start_service), and TWO cron jobs that both fire on the same `*/15` tick (etc/uci-defaults/91-singbox-ui-cron LINE_SUBS + LINE_RS) where subscription.uc:352 and nft-rulesets.uc both call `system([SINGBOX_INITD,'reload'])` = stop+start. Two stop+start cycles racing the same procd instance (or a cron reload racing a UI restart) can leave sing-box stopped, double-started, or apply nft against a half-written state. publish_atomic only guarantees the *config file* is never torn — it does not guarantee the daemon-restart sequence is atomic. No reviewer examined cross-process lifecycle serialization; the digest's CODE-net apply-lock finding is scoped to nft only.
- _Предложение:_ Wrap the service-mutating sequence (generate→check→procd restart→nft apply) in the same mkdir-based lock primitive already proven in nftables.uc, or gate all reload/restart callers through a single serialized init.d entry. At minimum, make the two cron lines share one wrapper script so they cannot both stop+start within the same tick. Add a VM regression that fires two concurrent `init.d reload` + one `ubus call singbox-ui restart` and asserts the service ends up running exactly once.

**2. procd respawn has no threshold/timeout/retry tuning**  
- _Почему важно:_ etc/init.d/singbox-ui:76 sets `procd_set_param respawn` bare. procd defaults (threshold 3600 / timeout 5 / retry 5) mean that if sing-box crashes >5 times in an hour (e.g. an OOM, a transient bind failure on a port still in TIME_WAIT after a racing restart, or a kernel-module-missing tproxy failure), procd gives up and the service stays DOWN with no auto-recovery until the next cron reload or manual intervention. The `sing-box check` gate prevents config-rejection loops but not runtime crashes. No reviewer looked at procd parameters.
- _Предложение:_ Set explicit `procd_set_param respawn <threshold> <timeout> <retry>` with a documented rationale, or add a watchdog/health note. Decide intentionally whether infinite respawn (retry 0) is desired for a proxy that must stay up.

**3. uci-defaults schema sentinel vs the separate 90-fwmark script — split-brain idempotency**  
- _Почему важно:_ 99-luci-singbox-ui guards ALL its migrations behind `CURRENT_SCHEMA=18` and exits early if cur_ver>=18 (lines 12-21), but 90-singbox-ui-fwmark (a SEPARATE uci-defaults file) has its own independent idempotency via per-key `uci get \|\| set` and its own `uci commit` (lines 16-26). The two are not coordinated: 90 commits the @global section *before* 99 runs, and 99's early-exit does not cover 90. If a future migration needs fwmark defaults present as a precondition this ordering is load-bearing but undocumented, and a partial-boot crash between 90's commit and 99's commit leaves _meta unset (schema 0) so 99 re-runs every migration on next boot — generally safe (each migration_*() is individually guarded) but the fakeip list→option migration (lines 34-56) is destructive (drops extra ranges with only a logger warning) and would re-log on every re-run. CODE/CI reviewers did not audit cross-script migration ordering or the partial-commit window.
- _Предложение:_ Document the 90-before-99 ordering contract in CLAUDE.md's migration section, and consider folding fwmark seeding into the schema-versioned 99 script so a single `uci commit` covers the whole migration set (the current split has two commit points = two crash windows).

**4. i18n has no CI hard-gate and the POT is regenerated by-hand only**  
- _Почему важно:_ There is a test_po_coverage.sh (±5 msgid tolerance, ≤5 untranslated) but it runs in the ucode/shell suite that is VM-gated; combined with the CI-tests finding that node-gated files don't run, the i18n guard's actual CI execution status is unverified. The POT (188 non-empty msgids) is regenerated only when a human runs scripts/regen-po.sh — nothing fails CI if a developer adds `_('New string')` and forgets to regen. Today ru sits at 2 untranslated (under threshold) but the margin is thin and CLAUDE.md itself flags ru as 'on the edge'.
- _Предложение:_ Confirm test_po_coverage.sh actually executes in CI (it's pure shell+grep+awk, so it could run in the host js-unit/lint job, not only the VM). Add a CI step that runs scripts/regen-po.sh and fails if it produces a non-empty `git diff` on the .pot — that catches forgotten regens deterministically.

**5. bbolt-client Rust: 29 unsafe/panic/unwrap sites in a no_std on-disk parser fed attacker-influenced data**  
- _Почему важно:_ bbolt-client/src/main.rs reads sing-box's cache.db (a bbolt file whose rule_set bucket content originates from remote subscription/rule-set servers) and has 29 unsafe/panic/unwrap/expect occurrences. A no_std raw-syscall binary parsing on-disk integers from a file influenced by a hostile rule-set server is exactly the place where an out-of-bounds slice or arithmetic overflow becomes a crash (or worse). The CODE-net reviewer covered curl/nft injection surfaces but no reviewer audited the Rust parser's bounds-checking against a malformed/adversarial cache.db. There are golden+stress+overflow+cyclic testdata fixtures, which is good, but the unwrap/panic density warrants a targeted read.
- _Предложение:_ Audit each unwrap/expect/unsafe in main.rs for whether the input is bounded before the index/cast; convert page-offset and length reads to checked_* arithmetic with graceful error returns rather than panic, since a panic in a no_std binary is an abort that surfaces to nft-rulesets.uc as an opaque non-zero exit.

**6. export_section.uc bare shebang is actually CORRECT, not a bug — but the consumer-coupling deserves a regression**  
- _Почему важно:_ The CLAUDEMD digest flags that CLAUDE.md documents export_section.uc as carrying a `-L` shebang while the file (line 1) has bare `#!/usr/bin/ucode`. That is doc-drift, not a code bug: rpcd invokes it as `ucode -L /usr/share/singbox-ui/lib export_section.uc` (explicit -L, per the file header comment), so the bare shebang is correct — same pattern as subscription.uc. The real gap is that nothing test-asserts export_section.uc is ALWAYS invoked with -L; if a future caller relies on the shebang it will silently fail to resolve `require('inbound')`.
- _Предложение:_ Fix CLAUDE.md to state export_section.uc is invoked via explicit -L (like subscription.uc), and add a one-line assertion to the rpcd prod-path test that `export_section` ubus method returns a resolved section (proving the -L wiring), not just status:ok.

**7. Disk-full / write-failure behavior of publish_atomic and the .prev backup churn**  
- _Почему важно:_ generate.uc publish_atomic (lines 164-203) writes config + last_state.json + (init.d) apply_state/service_state/pkg_version to /var/lib and /tmp. On a flash-storage OpenWrt device with a full overlay, fs.open('w') returns null and generate.uc exits(1) → init.d refuses to start (line 31 guard catches empty config, but a *successful-but-stale* prior config on a full disk means the user silently runs old config). The `.prev` backup is created on every generate (unlink+rename), doubling transient space use at the worst moment. No reviewer examined behavior under disk pressure on flash targets (the cache 'flash' storage mode writes cache.db to /etc/sing-box too).
- _Предложение:_ Log distinctly when generate.uc falls back to a stale config because the new write failed (vs. produced no config), so status_detail can surface 'config write failed, running previous'. Consider skipping the .prev copy when free space is low.
